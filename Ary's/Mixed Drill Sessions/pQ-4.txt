You are absolutely correct to question this. Your intuition is spot on.
**Yes, theoretically, a correctly implemented "Sliding Window" is self-cleaning.**
In a perfect world, if you increment `freq` when the window expands (`head` moves) and decrement `freq` when the window shrinks (`tail` moves), then by the time `tail` reaches the end of the array, every single increment should have been cancelled out by a decrement. `freq` should be all zeros, and `distinct` should be 0.
However, relying on this "automatic cleanup" is very risky in Competitive Programming. In your specific code, it failed (and caused issues for the next test case) because of a **hidden bug** that broke the cycle.
Here is the breakdown of why the "Auto-Reset" failed in your case
### 1. The "Hidden Bug" that broke the chain
The issue is here:
```cpp
    cin >> n >> m;
    set<int> st;
    for(int i=0; i<n;i++){ ... st.insert(x); } // Removes duplicates
    
    vector<int> arr(st.begin(),st.end()); // arr.size() might be LESS than n
    
    // THE BUG:
    while(tail < n) { ... } // You are looping up to 'n', not 'arr.size()'

```
**Scenario:**

* **Input:** `n=5`, numbers: `10 10 10 10 10`.
* **Set:** Removes duplicates. `arr` now has only **1 element**: `[10]`.
* **Loop:** Your loop runs `while(tail < 5)`.

**What happens inside the memory:**

1. When `head` or `tail` becomes `1`, you access `arr[1]`.
2. Since `arr` only has index `0`, `arr[1]` is **Out Of Bounds**.
3. C++ vectors do not stop you here. It reads whatever random garbage memory sits next to the vector. Let's say it reads the number `9999999`.
4. Your code calls `win_insert(9999999)`.
5. Inside `win_insert`, you try to access `divisors[9999999]`.
6. **CRASH (Segmentation Fault):** Your `divisors` array is only size `100010`. Accessing index `9999999` kills the program immediately.

Because the program crashed (or behaved unpredictably due to memory corruption) in Test Case 1, it never finished the cleanup. If the judge runs multiple test cases in the same process, the memory is now corrupted.

### 2. Why Manual Reset is "Best Practice"

Even if you fix the `n` vs `arr.size()` bug, relying on "Auto-Reset" is dangerous for these reasons:

1. **Early Breaks:** If you ever add an optimization like `if (ans == 0) break;` (common in these problems), you exit the loop early. The `tail` never finishes its job, `freq` is left dirty, and **Test Case 2 fails**.
2. **Exceptions/Logic Errors:** If you make a small logic mistake (like the one above), the cleanup fails.
3. **Safety:** Manually resetting at the start of `solve()` guarantees a clean slate, no matter how badly the previous test case crashed or broke.
